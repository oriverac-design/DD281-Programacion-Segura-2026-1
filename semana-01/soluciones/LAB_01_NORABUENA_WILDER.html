Ejercicio 4: Validación completa de contraseña robusta
public static boolean esContrasenaSegura(String pass) {
    if (pass == null || pass.length() < 8)  return false;
    boolean num  = pass.matches("[^0-9]*[0-9].*");
    boolean may  = pass.matches("[^A-Z]*[A-Z].*");
    boolean sim  = pass.matches(".*[!@#$%^&*].*");
    return num && may && sim;
}
💡 Pista: Propiedad de longitud de String — Método que evalúa si una cadena cumple una expresión regular.
Ejercicio 5: Validación de correo electrónico con expresión regular
import java.util.regex.*;

String correo = sc.nextLine();
Pattern patron = Pattern.compile(
    "^[\\w-\\.%+]+@[\\w.-]+\\.[a-zA-Z]{2,}$");
Matcher m = patron.matcher(correo);

if (m.matches()) {
    System.out.println("Correo válido");
} else {
    System.out.println("Correo inválido");
}
💡 Pista: Patrón: ^[\w._%+-]+@[\w.-]+\.[a-zA-Z]{2,}$  —  Método de Matcher que verifica coincidencia total.

Flujo principal1. El usuario ingresa su correo y contraseña.
2. El sistema valida las credenciales.
3. Se genera un token de sesión seguro.
4. El usuario accede al panel correspondiente según su rol.
Flujo alternativoSi las credenciales son incorrectas, el sistema muestra un mensaje de error.
Tras varios intentos fallidos, se bloquea temporalmente la cuenta.
Riesgo asociadoRobo de credenciales, ataques de fuerza bruta, suplantación de identidad
Medida de protecciónAutenticación multifactor (MFA), cifrado de contraseñas con algoritmos seguros (bcrypt), límite de intentos y registro de actividad sospechosa
Caso de Uso Seguro N° 2 — Registro de Nuevo Usuario
Actor(es)Usuario nuevo, sistema de registro
ObjetivoCrear una cuenta de usuario de forma segura y confiable
PrecondiciónEl usuario debe proporcionar datos válidos y únicos (correo no registrado)
Flujo principal1. El usuario completa el formulario de registro.
2. El sistema valida los datos ingresados.
3. Se envía correo de verificación.
4. El usuario confirma su cuenta y el sistema activa el perfil.
Flujo alternativoSi el correo ya existe, se muestra un mensaje de error.
Si los datos son inválidos, se solicita corrección antes de continuar.
Riesgo asociadoInyección de datos, creación de cuentas falsas, exposición de información sensible
Medida de protecciónValidación de entradas, verificación por correo electrónico, cifrado de datos personales y uso de CAPTCHA para evitar bots
Caso de Uso Seguro N° 3 — Procesamiento de Pago
Actor(es)Usuario comprador, sistema de pagos, pasarela de pago externa
ObjetivoRealizar el pago de productos de forma segura, garantizando la confidencialidad y la integridad de la transacción
Flujo principal1. El usuario selecciona los productos y procede al pago.
2. El sistema genera la orden y redirige a la pasarela de pago segura.
3. El usuario ingresa los datos de su tarjeta en el entorno cifrado.
4. La pasarela valida la transacción y devuelve la confirmación al sistema.
5. El sistema registra el pago y actualiza el inventario.
Riesgo asociadoIntercepción de datos financieros, fraude electrónico, manipulación de transacciones
Medida de protecciónUso de cifrado TLS/SSL, tokenización de datos de tarjeta, cumplimiento PCI DSS, monitoreo antifraude y validación de integridad de la transacción

Requerimientos
¿Qué preguntas de seguridad se deben hacer?
 Definir requisitos de seguridad y privacidad para datos sensibles. Analizar amenazas y riesgos asociados a las funcionalidades del sistema. Clasificar la información según niveles de confidencialidad y acceso.
Diseño
¿Qué patrones y controles arquitectónicos aplicar?
 Aplicar principios de diseño seguro: autenticación, autorización y cifrado. Modelar amenazas para identificar puntos vulnerables en el flujo de datos. Revisar la arquitectura y validar cumplimiento de estándares OWASP y PCI DSS.
Desarrollo
¿Qué buenas prácticas de codificación segura seguir?
 Implementar codificación segura evitando inyecciones y errores de validación. Gestionar dependencias y librerías actualizadas sin vulnerabilidades conocidas. Aplicar control de versiones seguro y auditoría de cambios críticos.
Pruebas
¿Qué tipos de pruebas de seguridad ejecutar?
 Ejecutar pruebas de penetración para validar resistencia ante ataques. Realizar análisis estático y dinámico del código para detectar vulnerabilidades. Validar autenticación, autorización y manejo de errores en los módulos críticos.
Despliegue
¿Qué configuraciones y controles aplicar en producción?
 Configurar el servidor con políticas seguras y deshabilitar servicios innecesarios. Gestionar credenciales mediante cifrado y rotación periódica de claves. Verificar integridad de componentes y aplicar firmas digitales antes del despliegue.
Monitoreo
¿Qué alertas e indicadores de seguridad implementar?
 Implementar alertas ante accesos sospechosos o fallos críticos. Registrar y auditar eventos de seguridad de forma continua. Analizar comportamiento de usuarios para detectar patrones anómalos.
